Velodrome

onActivityResult handlers for Android

One of the more horrible pieces of the Android lifecycle occurs inside onActivityResult. It inevitably descends into a mess of switch statements and spaghetti.

@Override
public void onActivityResult(int requestCode, int resultCode, Intent data) {
    if (resultCode != Activity.RESULT_OK) {
        return;
    }
    switch (requestCode) {
        case EDIT_TEXT:
            Log.d(TAG, "EditText dialog returned " + data.getStringExtra("value"));
            break;
        case DRILL_ACTIVITY:
            Log.d(TAG, "Some drilldown Activity returned" + data.getParcelableExtra("object"));
            break;
        case DELETE_RESULT:
            Log.d(TAG, "Deletion successful");
            break;
        case RETICULATING_SPLINES:
            Log.d(TAG, "Splines reticulated");
            break;
        default:
            Log.d(TAG, "Dunno");
    }
}

A more reasonable approach is to encapsulate each piece of callback behavior into its own handler method. That is precisely what Velodrome does.

// In your fragment or activity class.

@HandleResult(0)
public void onActivityReturn(Intent data) {
    Log.d("Velo", data.getStringExtra("text"));
}

@HandleResults({2, 3})
public void onMultipleCodes(Intent data) {
    Log.d("Velo", "one of two possible things just happened.");
}

@HandleResult(value = 1, resultCode = Activity.RESULT_CANCELED)
public void onDialogCancel(Intent data) {
    Log.d("Velo", "Canceled");
}

@Override
public void onActivityResult(int requestCode, int resultCode, Intent data) {
    super.onActivityResult(requestCode, resultCode, data);
    Velodrome.handleResult(this, requestCode, resultCode, data);
}

Install

dependencies {
    compile 'com.levelmoney.velodrome:velodrome:0.9-SNAPSHOT@aar'
}

How it works

Velodrome currently discovers handlers via simple reflection. All handlers must be declared public.
